Your agent's safety net
is a .md file.

Agent skills are markdown files. The model reads them, follows the instructions, and calls tools. There is nothing between "the skill says don't do that" and the model doing it anyway.

"Most failures are not fancy exploits — they're someone messaged the bot and the bot did what they asked." — OpenClaw safety documentation

Skills are just prompts

A SKILL.md is natural language instructions the model reads and hopefully follows. "Run tests before deploying" is a suggestion, not a constraint. Nothing enforces it.

Deny-lists are never complete

You block rm -rf but miss find -delete. You block DROP TABLE but miss TRUNCATE. The agent finds the gap you didn't think of.

There's no execution layer

Skills have no runtime. The model reads markdown, decides what to do, and calls tools directly. There's no gate, no interceptor, no enforcement between intent and action.

More skills = more trust

Every new skill hands the model more tools and more instructions to follow correctly. You're scaling capability while scaling risk — with nothing but the prompt standing between them.

Your skills are prompts.
Make them enforceable.

PetriFlow adds an enforcement layer below your skills. The skill tells the agent what to do. The rule makes sure it can't do it wrong.

Skill alone (hope-based)

Delete unused files 3. Verify cleanup # the model reads this. # the model might follow it. # nothing makes it.

# same SKILL.md as before # same instructions, same tools # but now, add rules: map bash.command cp as backup map bash.command rm as delete require backup before delete # the model can try to skip the backup. # the rm call is intercepted and blocked # before it reaches the shell.

The skill says what to do. The rule ensures it. No code path can bypass it.

Real agents. Real constraints.
Mathematically proven.

Four production safety rules. Three require zero human approval — safety comes entirely from the rule structure.

observe-before-send

Don't message blindly

The agent must read a channel before it can send a message to it. Each read earns one send. Channel-matched, so reading #general doesn't unlock #production.

✓ Impossible to send without reading the channel first. 1:1 ratio enforced structurally.
read #deploys → send #deploys → send #deploys BLOCKED (must read again)
test-before-deploy

Don't ship broken code

Can't deploy to staging without passing tests. Can't go to production without staging. Production requires a human to approve.

✓ Production deploy is a manual action. No amount of prompt injection can auto-ship.
test → build → stage → ship BLOCKED (needs human)
research-before-share

Don't share fabrications

Each web fetch earns one share token. Each share consumes one. The agent can't broadcast "findings" without having actually fetched real sources.

✓ 1:1 fetch-to-share ratio. No research means no sharing is possible.
share BLOCKED → fetch url → share → share BLOCKED (must fetch again)
backup-before-delete

Don't destroy without a safety net

Every destructive command is blocked until a backup covers the target path. git stash covers everything. cp src/ bak/ covers src/.

✓ Backup must actually succeed (deferred check). Path coverage validated semantically.
rm -rf data/ BLOCKED → cp -r data/ /tmp/bak → rm -rf data/

Formal verification.
Not just testing.

Every safety rule compiles down to a formally verifiable state machine. Before your agent runs, PetriFlow enumerates every reachable state and checks your invariants across all of them.

This isn't sampling. It's not fuzzing. It's exhaustive proof. If the verifier says "safe," it means safe in every possible execution, not just the ones you tested.
